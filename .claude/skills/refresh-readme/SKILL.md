# refresh-readme

The README has two auto-maintained blocks:

- **Current State** — between `<!-- CURRENT_STATE:START -->` and `<!-- CURRENT_STATE:END -->`. Shows three screenshots (map, gambit editor, combat) plus a Claude-authored prose description of what the game currently does, how to play it, and the dev surface (debug pages, editable content, etc.).
- **Chassis** — between `<!-- CHASSIS:START -->` and `<!-- CHASSIS:END -->`. Shows one screenshot per chassis arranged in a 2-column HTML table. Each card image already renders the attack stats (name / DMG / CD / Init), so the block does not emit a separate Markdown stats table.

Both blocks are regenerated by overwriting files in place — no stale-file sweep. Any actual cleanup is a **logical drift audit** described below.

## Source files this skill touches

| File | Role |
|---|---|
| `README.md` | Host for both auto-maintained blocks. |
| `src/ui/screens/ChassisPreview/ChassisPreview.tsx` | Preview page at `/?preview=chassis` — edit `CHASSIS_ENTRIES` here when adding/removing chassis. |
| `src/content/attacks.json` | Attack stats rendered inside each chassis card image. |
| `src/logic/state/types.ts` | The `Chassis` union — source of truth for which chassis exist. |
| `tests/ui/ChassisPreview.test.tsx` | Drift guard for the Chassis preview contract. |
| `tests/e2e/chassis-preview.spec.ts` | Screenshots each chassis card into `test-results/artifacts/chassis/<id>.png` and generates `chassis-overview.md`. Edit `CHASSIS_ORDER` / `buildMarkdown` when the rendered shape must change. |
| `tests/e2e/readme-screenshots.spec.ts` | Generates `test-results/artifacts/readme/{map,editor,combat}.png`. |
| `doc/screenshots/chassis/*.png` | Committed per-chassis card screenshots (one file per chassis). |
| `doc/screenshots/readme/*.png` | Committed Current State screenshots. |
| `doc/generated/chassis-overview.md` | Generated chassis Markdown block (2-col HTML table of `<img>` refs into `doc/screenshots/chassis/`). |
| `doc/roadmap.md` | Read-only input for the Current State description — reflects what ships next. |

## Workflow

Run the whole workflow as one pass. Each section has its own drift audit, its own confirmation step, and its own regeneration step. When both sections are covered, finish with `pnpm check`.

### 1. Current State — drift audit

Goal: decide whether the Current State description and screenshots still match what the game actually does.

Read:
- `doc/architecture.md` §0 (Quick Start — lists what is currently built).
- `doc/roadmap.md` (what landed, what is next).
- Recent commit subjects (`git log --oneline -20`).
- `src/ui/App.tsx` — the authoritative list of screens and debug pages.
- The current `README.md` block between `<!-- CURRENT_STATE:START -->` / `<!-- CURRENT_STATE:END -->`.

Ask yourself:
- Does the prose still describe the current state? New screens, new mechanics, removed features?
- Are the "How to play" steps still correct end-to-end?
- Is the "Debug pages" list still accurate (check `App.tsx` conditionals)?
- Should the three screenshots still be map / editor / combat, or has the set changed (new key screen, removed screen)?
- Is the heading still right (current code may have moved past the last version referenced)?

Produce a short diff of proposed changes vs. the current block, and **stop for user approval.** If nothing changed since the last refresh and screenshots can stay, say so and still proceed to regeneration (screenshots may drift even when prose doesn't).

### 2. Current State — regenerate

1. Run:
   ```bash
   pnpm refresh-readme-artifacts
   ```
   This runs both artifact-generating e2e specs and copies the outputs into `doc/screenshots/readme/{map,editor,combat}.png`, `doc/screenshots/chassis/<id>.png` (one per chassis), and `doc/generated/chassis-overview.md`. The specs themselves write only to the gitignored `test-results/artifacts/` scratch dir — regular `pnpm check` runs never dirty these paths. This one script covers the regeneration step for both the Current State and Chassis sections (see §4).
2. If the drift audit identified that a screenshot needs a different scene (e.g., a newly added screen), update `readme-screenshots.spec.ts` first, then rerun.
3. Read each new screenshot with the Read tool to confirm it renders correctly.
4. Draft the new Current State prose (Claude-authored, not generated by a tool). Keep the structure: short lead sentence, three screen blurbs each followed by their screenshot, a "How to play" numbered list, and any additional callouts (squad persistence, editable content, debug pages). Keep the `<sub>…run /refresh-readme…</sub>` trailer line.
5. Rewrite everything between `<!-- CURRENT_STATE:START -->` and `<!-- CURRENT_STATE:END -->` in `README.md`. Leave the markers untouched.

### 3. Chassis — drift audit

Goal: decide whether the chassis preview's scope still matches the project.

Read:
- `src/logic/state/types.ts` (the `Chassis` union).
- `src/content/attacks.json`.
- `src/content/player-squad.json` and any other chassis-adjacent content that has landed since.
- `src/ui/screens/ChassisPreview/ChassisPreview.tsx` (`CHASSIS_ENTRIES`).
- `tests/e2e/chassis-preview.spec.ts` (`CHASSIS_ORDER`, `buildMarkdown`).
- Roadmap entries in `doc/roadmap.md` that mention chassis or stats.

Ask yourself:
- Is every member of the `Chassis` union represented in `CHASSIS_ENTRIES` and `CHASSIS_ORDER`? Any extras that no longer belong?
- Does the in-card layout (attack name / DMG / CD / Init) still reflect the most useful chassis stats, or has the project grown stats worth adding (e.g., HP, speed, module slots)? Changes here go into `ChassisPreview.tsx` so the card image itself updates.
- Are there per-chassis stats that now live somewhere other than `attacks.json` and should be surfaced on the card?
- Is the 2-column HTML-table grid still the right layout (e.g., does the roster now warrant 3 columns)?

Produce a short drift report and **stop for user approval** before changing any code.

### 4. Chassis — apply approved changes + regenerate

If drift was approved, edit in this order so the drift guard test stays honest:
1. Update `CHASSIS_ENTRIES` in `ChassisPreview.tsx` and any new stat columns.
2. Update `CHASSIS_ORDER` and `buildMarkdown` in `tests/e2e/chassis-preview.spec.ts`.
3. Update `tests/ui/ChassisPreview.test.tsx` if the contract shape changed.
4. `pnpm test tests/ui/ChassisPreview.test.tsx` — must pass before regenerating.

Then regenerate. If you haven't already run `pnpm refresh-readme-artifacts` in §2, run it now — it regenerates the per-chassis screenshots and Markdown alongside the Current State screenshots:
```bash
pnpm refresh-readme-artifacts
```
Overwrites `doc/screenshots/chassis/<id>.png` (one per chassis) and `doc/generated/chassis-overview.md`. Spot-check at least one newly-regenerated card image with the Read tool to verify it renders correctly.

Finally replace everything between `<!-- CHASSIS:START -->` and `<!-- CHASSIS:END -->` in `README.md` with the contents of `doc/generated/chassis-overview.md`. Keep the markers untouched.

### 5. Final check

Run `pnpm check` per `CLAUDE.md`. Fix any regression before reporting done.

### 6. Report

Summarise in 1–3 sentences per section: what drift (if any) was found and fixed, that screenshots/markdown were regenerated, and that `pnpm check` passed.

## Notes

- Do not run standalone Node scripts — all execution goes through `pnpm` scripts (`pnpm test`, `pnpm e2e`, `pnpm check`) per `CLAUDE.md`.
- The README blocks are auto-maintained; never hand-edit them — edit the source (content JSON, React components, Playwright specs, or this skill's prose drafting) and rerun this skill.
- If the drift audit surfaces a decision that affects design docs (`doc/gameplay.md`, `doc/architecture.md`, etc.), update those in the same step — code and docs must not drift.
- Old versioned screenshots under `doc/screenshots/` (e.g., `combat-v0.4.png`) that are no longer referenced by the README are *not* deleted automatically. If the drift audit notices unreferenced legacy screenshots, surface them to the user as a cleanup suggestion rather than deleting silently.
